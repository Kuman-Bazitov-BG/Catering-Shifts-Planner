Authentication

Call POST /api/auth/login with your email and password. On success you receive a JWT token (expires in 7 days). Send the token on every subsequent request as an HTTP header: 

Authorization: Bearer <your-jwt-token>

Missing, malformed, or expired tokens always return 401 Unauthorized with { "error": "Unauthorized." }.

The mobile app must store the token in secure device storage and attach it to every API call. There is no session cookie — the REST API is entirely stateless.

Error format

All error responses have the same JSON shape:

{ "error": "<human-readable message>" }
HTTP statusMeaning
400Bad request — missing or invalid input
401Unauthorized — missing or invalid Bearer token
403Forbidden — authenticated but not a member of the group
404Not found — shift does not exist
409Conflict — business rule violation (e.g. shift not active, already joined)

isJoined: true when the authenticated user has joined.
extraSlots: null when not joined; 0–3 otherwise.

POST /api/shifts/{id}/slots Bearer required

Set the number of extra slots the authenticated user is reserving on top of their own seat. Must have joined the shift first. The value is clamped to 0–3 regardless of what is sent. Each extra slot counts toward the shift's staffCount and capacity.

Path parameters

ParamTypeDescription
idintegerShift ID

Request body

FieldTypeRequiredNotes
extraSlotsnumber✓Desired count; clamped to 0–3
{ "extraSlots": 2 }

Response 200 OK

{ "ok": true }

Extra slots capacity math

If you reserve extraSlots: 2 you bring 2 extra people. Your contribution to staffCount becomes 3 (yourself + 2). The shift's capacity is compared against the total staffCount. Full capacity does not block joining — staff resolve over-capacity manually.

Error responses

StatusWhen
400Non-integer shift ID or non-numeric extraSlots
400Invalid JSON body
401Missing or invalid Bearer token
403User is not a member of the group
404Shift does not exist
409User has not joined the shift yet, or shift is not active
